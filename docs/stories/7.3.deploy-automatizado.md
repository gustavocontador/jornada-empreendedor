## Story

**As a** developer,
**I want** automated deployment to production when pushing to main,
**so that** features are delivered to users quickly and reliably.

## Acceptance Criteria

1. [ ] Push em `main` → trigger deploy production automaticamente
2. [ ] Push em `develop` → trigger deploy staging automaticamente
3. [ ] Frontend: Vercel auto-deploy configurado (via vercel.json)
4. [ ] Backend: Railway auto-deploy configurado (via railway.json)
5. [ ] Database: Supabase migrations auto-executadas (via CLI)
6. [ ] Variáveis de ambiente configuradas em cada plataforma (Railway, Vercel, Supabase)
7. [ ] Health checks validados após deploy (rollback se falhar)
8. [ ] URLs de produção documentadas no README.md
9. [ ] Deploy time < 3min (main → production live)
10. [ ] Rollback automático funcional (se health check falhar)

### Task 1: Setup Vercel (Frontend) (AC: 3)
- [ ] Criar conta Vercel (se não existir)
- [ ] Instalar Vercel CLI: `npm i -g vercel`
- [ ] Autenticar: `vercel login`
- [ ] Criar `frontend/vercel.json`:
  ```json
  {
    "buildCommand": "npm run build",
    "outputDirectory": ".next",
    "framework": "nextjs",
    "installCommand": "npm ci",
    "env": {
      "NEXT_PUBLIC_API_URL": "@api_url",
      "NEXT_PUBLIC_SENTRY_DSN": "@sentry_dsn_frontend",
      "NEXT_PUBLIC_ENVIRONMENT": "@environment"
    },
    "headers": [
      {
        "source": "/(.*)",
        "headers": [
          {"key": "X-Frame-Options", "value": "DENY"},
          {"key": "X-Content-Type-Options", "value": "nosniff"}
        ]
      }
    ]
  }
  ```
- [ ] Conectar repositório GitHub ao Vercel
- [ ] Configurar branches: `main` → production, `develop` → preview
- [ ] Configurar env vars no Vercel Dashboard

### Task 2: Setup Railway (Backend) (AC: 4)
- [ ] Criar conta Railway (se não existir)
- [ ] Instalar Railway CLI: `npm i -g @railway/cli`
- [ ] Autenticar: `railway login`
- [ ] Criar `backend/railway.json`:
  ```json
  {
    "build": {
      "builder": "NIXPACKS",
      "buildCommand": "pip install -r requirements.txt"
    },
    "deploy": {
      "startCommand": "uvicorn app.main:app --host 0.0.0.0 --port $PORT --workers 4",
      "healthcheckPath": "/api/v1/health",
      "healthcheckTimeout": 30,
      "restartPolicyType": "ON_FAILURE",
      "restartPolicyMaxRetries": 3
    }
  }
  ```
- [ ] Criar `backend/nixpacks.toml`:
  ```toml
  [phases.setup]
  nixPkgs = ['python312', 'postgresql']

  [phases.install]
  cmds = ['pip install -r requirements.txt']

  [start]
  cmd = 'uvicorn app.main:app --host 0.0.0.0 --port $PORT --workers 4'
  ```
- [ ] Conectar repositório GitHub ao Railway
- [ ] Configurar env vars no Railway:
  - `DATABASE_URL` (Supabase connection string)
  - `JWT_PRIVATE_KEY`, `JWT_PUBLIC_KEY`
  - `SENTRY_DSN`, `SENTRY_ENVIRONMENT`

### Task 3: Setup Supabase (Database) (AC: 5)
- [ ] Criar projeto no Supabase Dashboard
- [ ] Obter connection string (Settings → Database)
- [ ] Instalar Supabase CLI:
  ```bash
  npm install supabase --save-dev
  ```
- [ ] Inicializar: `npx supabase init`
- [ ] Criar migration inicial:
  ```bash
  npx supabase migration new initial_schema
  ```
- [ ] Aplicar migrations localmente:
  ```bash
  npx supabase db push
  ```
- [ ] Configurar auto-migration no CI (GitHub Actions):
  ```yaml
  - name: Run Supabase Migrations
    run: npx supabase db push
    env:
      SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
      SUPABASE_DB_URL: ${{ secrets.SUPABASE_DB_URL }}
  ```

### Task 6: GitHub Actions - Deploy Workflow (AC: 1, 2)
- [ ] Atualizar `.github/workflows/ci.yml` com job de deploy:
  ```yaml
  deploy-production:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: [lint-backend, lint-frontend, test-backend, test-frontend]
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    steps:
      - uses: actions/checkout@v4

      - name: Deploy Frontend to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
          working-directory: frontend

      - name: Deploy Backend to Railway
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
        run: |
          npm install -g @railway/cli
          railway up --service backend --environment production

      - name: Run Supabase Migrations
        run: npx supabase db push
        env:
          SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
          SUPABASE_DB_URL: ${{ secrets.SUPABASE_DB_URL }}
  ```

## Dev Notes

### Contexto do Projeto

Este story finaliza a infraestrutura base (Epic 7) permitindo deploy automatizado do Sistema V2. Após implementação:
- **Velocidade:** Deploy em < 3min (push → production)
- **Confiabilidade:** Rollback automático se health check falhar
- **Observabilidade:** Sentry capturando erros desde deploy inicial

### Arquitetura de Deploy

```
GitHub (main branch)
    │
    ├─► Vercel (Frontend)
    │   ├── Next.js 15 build
    │   ├── Edge CDN
    │   └── URL: jornada-empreendedor.vercel.app
    │
    ├─► Railway (Backend)
    │   ├── FastAPI (Uvicorn)
    │   ├── Health checks
    │   └── URL: api.jornada-empreendedor.railway.app
    │
    └─► Supabase (Database)
        ├── PostgreSQL 16+ with RLS
        ├── Storage buckets
        └── Auto-migrations via CLI
```

### Dependências de Stories Anteriores

- **Story 7.1 (Monorepo):** Workspaces configurados
- **Story 7.2 (CI):** Quality gates bloqueiam deploy se testes falharem

### Referências Técnicas

- **Vercel Docs:** https://vercel.com/docs
- **Railway Docs:** https://docs.railway.app
- **Supabase CLI:** https://supabase.com/docs/guides/cli
- **Nixpacks:** https://nixpacks.com/docs

### Configurações Críticas

**Railway Healthcheck:**
- Path: `/api/v1/health`
- Timeout: 30s
- Retry: 3x
- Action on failure: Rollback to previous deployment

**Vercel Headers (Security):**
- X-Frame-Options: DENY (previne clickjacking)
- X-Content-Type-Options: nosniff (previne MIME sniffing)

### Observabilidade (Fase 1 integrada)

A Fase 1 já implementou:
- `/api/v1/health` (health check principal)
- `/api/v1/health/ready` (Kubernetes readiness probe)
- `/api/v1/health/live` (Kubernetes liveness probe)
- Performance middleware (X-Response-Time header)

Esses endpoints são usados pelo Railway para validar deploy.
